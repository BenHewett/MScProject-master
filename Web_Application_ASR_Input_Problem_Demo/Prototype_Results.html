The "Input" Problem (6)

Recap: User Feedback

Prototype #1: Key Conclusions

Pause-threshold Increase

Users still felt pressured which caused further disfluency

Poor user experience when fluency achieved due to long wait times

Processing time: the longer the pause threshold the longer the processing time

Prototype #2: Key Conclusions

Recording

Sense of time pressure alleviated due to increased user control

Some usability issues regarding button placement

Processing time: elongated disfluency captured within audio resulting in processing time remaining high

Prototype #3: Key Conclusions

Optional Pausing/Restarting

Sense of time pressure further alleviated due to fine-grained user control

Some usability issues regarding number of buttons

Some usability issues regarding the learning curve

Processing time: possible reduction in captured disfluency due to user control, if utilised

Prototype #4: Key Conclusions

Optional Pausing/Restarting + Resetting

Sense of control goes beyond the bounds of being useful and into the realm of being confusing

Significant usability problems regarding ease of use and practical usability

Learning curve too steep

Processing time: possible reduction in captured disfluency due to user control if user used reset feature repeatedly to capture fluency

Overall: Key Conclusions

Prototype #1: not feasible due to not solving the problem of time-pressure and incurring problematic processing time

Prototype #2: potential feasibility due to solving the time-pressure problem but processing time could still be a high cost

Prototype #3: feasible due to solving the time-pressure problem and potentially reducing cost of processing time

Prototype #4: not feasible due to user confusion/usability issues
